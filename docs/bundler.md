# Bundler — architecture and operation

> Node.js/TypeScript server that receives UserOperations over JSON-RPC and submits them to the
> real EntryPoint via a `handleOps` transaction. Also exposes `POST /deploy` (SETUP) and serves
> the built frontend as static files. Overall flow: [architecture.md](./architecture.md).

## Positioning — a MINIMAL bundler (pedagogical)

Deliberately minimal, built for learning: 1 UserOp per bundle, no mempool, no gas estimation, no
[ERC-7562](https://eips.ethereum.org/EIPS/eip-7562) validation (the spec banning certain opcodes
during simulation to stop mempool DoS — see [limitations.md](./limitations.md)), synchronous
submission. **Not a production bundler.** Reference: [eth-infinitism/bundler](https://github.com/eth-infinitism/bundler),
[ERC-4337 spec](https://eips.ethereum.org/EIPS/eip-4337). Uses **[viem](https://viem.sh)** for all
Ethereum interaction.

## File breakdown (`bundler/src/`)

| File | Responsibility |
|---|---|
| `loadEnv.ts` / `config.ts` | Load `bundler/.env`; create `publicClient` (read) + `walletClient` (bundler account) |
| `entryPoint.ts` | Partial EntryPoint ABI (`getUserOpHash`, `handleOps`, `balanceOf`, `getNonce`) |
| `userOp.ts` | `PackedUserOperation` type, field packing, hex ↔ bigint serialization |
| `handler.ts` | `eth_sendUserOperation`: validate → dry-run simulate → submit → wait for receipt |
| `deploy.ts` | `POST /deploy` logic: deploys a `SecretQuestionAccount` for a client-derived signer address |
| `secretQuestionArtifact.ts` | ABI + creation bytecode, embedded so `/deploy` works without Foundry (generated by [`scripts/gen-artifact.js`](../bundler/scripts/gen-artifact.js)) |
| `server.ts` | Express: `POST /rpc`, `POST /deploy`, CORS, `GET /*` static frontend when `frontend/dist` exists |
| `index.ts` | Server startup |

## Flow of an `eth_sendUserOperation` request

```
browser                               bundler (server → handler)         EntryPoint (Sepolia)
────────                              ──────────────────────────         ───────────────────
 1. getNonce() ───────────────────────────────────────────────────────▶ read
 2. build callData = execute(counter, 0, increment())
 3. getUserOpHash(userOp) ─────────────────────────────────────────────▶ read (canonical hash)
 4. signMessage({ raw: hash })  (personal_sign)
 5. POST eth_sendUserOperation ──────▶ validate shape + re-check hash ──▶ read
                                       simulateContract(handleOps) ─────▶ eth_call DRY-RUN (no gas, no tx)
                                       handleOps([op], bundler) ────────▶ TX — only if dry-run passed
 6. ◀──────────── { result: { userOpHash, txHash } }
```

## Key technical points

- **`userOpHash`**: never recomputed by hand — `entryPoint.getUserOpHash(userOp)` (a view)
  returns the canonical hash (embeds an EIP-712 domain in v0.8). The client signs it with
  `personal_sign`; the contract applies the same EIP-191 prefix.
- **Serialization**: JSON has no `bigint`, so wire fields are hex strings (`UserOperationHex`);
  `toPacked()`/`toHexOp()` convert to/from the internal `PackedUserOperation`. Packed fields
  (`accountGasLimits`, `gasFees`, `paymasterAndData`) are bit-shifted by hand in `userOp.ts` — a
  packing mistake doesn't crash, it silently produces an invalid UserOp.
- **The `eth_call` dry-run** (`handler.ts`): before the real `handleOps` tx, the bundler
  `simulateContract`s it — free, no state change. A UserOp that would revert (bad signature →
  `AA24`, underfunded paymaster → `AA31` — `AA##` is the EntryPoint's own revert-reason numbering)
  is rejected over JSON-RPC, no transaction ever sent. This matters because a wrong
  secret-question answer produces an invalid signature on every guess; without the dry-run each
  wrong guess would cost real sponsored gas. Does **not** replace ERC-7562 mempool validation.
- **Who signs what**: the account's signer signs the **UserOp** — derived in the browser from
  the secret answers, never sent to the bundler (only the signature is). The **bundler** signs
  the `handleOps` **transaction** (`BUNDLER_PRIVATE_KEY`) and is the `beneficiary` reimbursed by
  the Paymaster. The **deployer** (`DEPLOYER_PRIVATE_KEY`) signs `SecretQuestionAccount`
  deployments for `POST /deploy` — deliberately a different key from the signer, so the server
  never learns the user's derived private key.

## Exposed endpoints

| Method / route | Behavior |
|---|---|
| `POST /rpc` — `eth_sendUserOperation` | Params `[userOp, entryPoint]`, returns `{ userOpHash, txHash }` |
| `POST /rpc` — `eth_supportedEntryPoints` / `eth_chainId` | Returns `[configured entryPoint]` / `0xaa36a7` |
| `POST /deploy` | REST, not JSON-RPC. Body `{ signerAddress }` → deploys a `SecretQuestionAccount`, returns `{ account }` |
| `GET /*` | Serves the built frontend (`frontend/dist`) when present, SPA fallback to `index.html` |

## Limitations

No gas estimation, no ERC-7562 validation, a single UserOp per bundle, synchronous submission
(the dry-run helps but doesn't remove these). Details: [limitations.md](./limitations.md).
